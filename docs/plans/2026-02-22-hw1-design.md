# HW1 Design: Training a DNN using PyTorch

## Context

Homework 1 asks us to train three models that predict the outcome of a robot push action on an object. The simulation (MuJoCo + UR5e robot arm) acts as a physics oracle: it runs the full rigid-body dynamics and we sample its outputs as supervised training data. Once trained, the models predict outcomes without running the simulation.

The object always spawns at a fixed position at the centre of the table `(0.6, 0.0)`. All four push actions are designed to pass through that point, so contact is guaranteed every episode. The only randomness is the object's type (cube or sphere, 50/50) and size (uniform 2–3 cm). This makes the ML task clean: the model must learn how visual appearance encodes shape and size, and how those interact with each push direction.

## Deliverables

| # | Model | Input | Output |
|---|-------|-------|--------|
| 1 | MLP | flattened image + one-hot action | (x, y) object position after push |
| 2 | CNN | 128×128 image + one-hot action | (x, y) object position after push |
| 3 | CNN Encoder-Decoder | 128×128 image + one-hot action | 128×128 reconstructed post-push image |

## Approach

**Single notebook** (`src/hw1.ipynb`) covering data loading, all three models, training loops, and visualisation. Data collection is handled by a fixed version of `homework1.py`.

## Data Collection

The existing `collect()` function in `homework1.py` has a bug: it captures only the *after* state, but training requires both before and after. The corrected loop is:

```
reset() → state() → img_before → step(action) → state() → pos_after, img_after
```

4 processes × 250 samples = **1000 samples total**. Each process saves:

- `imgs_before_{idx}.pt` — `(250, 3, 128, 128)` uint8
- `imgs_after_{idx}.pt`  — `(250, 3, 128, 128)` uint8
- `positions_{idx}.pt`   — `(250, 2)` float32
- `actions_{idx}.pt`     — `(250,)` uint8

The notebook loads and concatenates all four shards. Preprocessing: images ÷ 255 → `[0, 1]` float, actions one-hot encoded to length-4, 80/20 train/val split.

## Model Architectures

### Model 1 — MLP (position prediction)

```
Input: flatten(img_before/255) → (49152,) ++ one_hot(action) → (49156,)
Linear(49156 → 1024) → ReLU
Linear(1024 → 512)   → ReLU
Linear(512 → 2)                ← predicted (x, y)
Loss: MSE
```

Intentionally naive — no spatial awareness. Pixels treated as an unordered list. Serves as the baseline to show why CNNs are better.

### Model 2 — CNN (position prediction)

```
Conv2d(3→32,  4×4 stride 2) → ReLU    128×128 → 64×64
Conv2d(32→64, 4×4 stride 2) → ReLU     64×64  → 32×32
Conv2d(64→128,4×4 stride 2) → ReLU     32×32  → 16×16
Conv2d(128→256,4×4 stride 2)→ ReLU     16×16  →  8×8
AdaptiveAvgPool2d(1) → flatten → (256,)
concat one_hot(action, 4)    → (260,)
Linear(260 → 128) → ReLU
Linear(128 → 2)                         ← predicted (x, y)
Loss: MSE
```

Action is injected after the conv backbone so the convolutional layers learn a general visual representation of the object; the action then biases the regression head toward the correct directional outcome.

### Model 3 — CNN Encoder-Decoder (image reconstruction)

```
Encoder:
  Conv2d(3→32,   4×4 stride 2) → ReLU    128 → 64
  Conv2d(32→64,  4×4 stride 2) → ReLU     64 → 32
  Conv2d(64→128, 4×4 stride 2) → ReLU     32 → 16
  Conv2d(128→256,4×4 stride 2) → ReLU     16 →  8
  Conv2d(256→256,4×4 stride 2) → ReLU      8 →  4   ← bottleneck (256, 4, 4)

Action injection:
  broadcast one_hot(action, 4) → (4, 4, 4)
  concat with bottleneck       → (260, 4, 4)

Decoder:
  ConvTranspose2d(260→256, 4×4 stride 2) → ReLU     4 →  8
  ConvTranspose2d(256→128, 4×4 stride 2) → ReLU     8 → 16
  ConvTranspose2d(128→64,  4×4 stride 2) → ReLU    16 → 32
  ConvTranspose2d(64→32,   4×4 stride 2) → ReLU    32 → 64
  ConvTranspose2d(32→3,    4×4 stride 2) → Sigmoid 64 → 128   ← reconstructed image [0,1]
Loss: MSE on pixels
```

Action is injected at the bottleneck — the most compressed representation of the scene — so the decoder reconstructs the scene conditioned on which push was applied.

## Training Configuration

| Setting | Value |
|---------|-------|
| Optimiser | Adam |
| Learning rate | 1e-3 |
| Batch size | 64 |
| Epochs (pos models) | 50 |
| Epochs (image model) | 100 |
| Device | MPS → CPU fallback |

## Notebook Structure

1. **Data collection** — fix `collect()`, run 4 processes, merge shards
2. **Dataset & DataLoader** — shared preprocessing, train/val split
3. **MLP** — define, train, plot loss curve, scatter predicted vs actual positions
4. **CNN** — same evaluation as MLP, compare MSE
5. **Encoder-Decoder** — train, display `[img_before | img_predicted | img_after]` triplets
